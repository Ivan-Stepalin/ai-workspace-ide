# AI Workspace IDE

Браузерный **пульт управления агентами** на базе Claude над набором проектов. Главный интерфейс — **нативный чат с агентом** (пузыри сообщений, markdown с подсветкой кода, видимые вызовы инструментов, токеновый стриминг); рядом — вспомогательные **bash-терминалы** и **git** (log / commit / push). **Кода в браузере не редактируем и не просматриваем** — агент сам правит файлы в своей рабочей директории через инструменты `claude`.

**Каждый проект открывается в своей вкладке браузера**: на входе — лаунчер выбора проекта, выбранный проект живёт в URL (`?p=<id>`), другой проект открывается в новой вкладке. **Сессии чатов и терминалов живут на сервере** и переподключаются при повторном открытии проекта. Есть **общий менеджер** — кросс-проектный воркспейс, который видит все проекты и добавляет репозитории по ссылке. Вход — по логину; права **пер-пользовательские** (роль — лишь шаблон), а пользователями управляет **сисадмин** через отдельную страницу. Фронтенд — устанавливаемое PWA, опционально доступен Telegram-бот к тем же агентам.

```
ai-workspace-ide/
├── backend/    # Node + Express + WebSocket: проекты, git, авторизация, чат и bash-терминалы (claude CLI), Telegram-бот
└── frontend/   # React + Vite + CSS Modules: интерфейс пульта, PWA
```

Пользовательские данные (проекты и БД) хранятся **вне** репозитория — см. `DATA_DIR` ниже.

## Требования

- **Node.js 20+**
- **[Claude CLI](https://docs.claude.com/claude-code)** в `PATH` — бэкенд вызывает `claude` для агентов (headless `claude -p --output-format stream-json` для чата и Telegram).
- **git** — для clone/commit/push.
- (опционально) **mkcert** — доверенный локальный HTTPS, нужен только чтобы открывать/устанавливать PWA с телефона.

## Установка

```bash
git clone git@github.com:Ivan-Stepalin/ai-workspace-ide.git
cd ai-workspace-ide

npm run install:all
# то же вручную:
#   npm --prefix backend install
#   npm --prefix frontend install
```

## Настройка окружения

```bash
cp backend/.env.example backend/.env
cp frontend/.env.example frontend/.env
```

- **backend/.env**
  - `PORT` — порт бэкенда (HTTP + WebSocket), по умолчанию `3001`.
  - `DATA_DIR` — где хранить данные (создаются `projects/` и `workspace.db`), по умолчанию `./data`.
    Можно переопределить точечно: `PROJECTS_DIR`, `DB_PATH`.
  - **Авторизация:**
    - `AUTH_SECRET` — секрет для подписи сессионного токена (httpOnly-cookie). В проде задать обязательно
      (сгенерировать: `node -e "console.log(require('crypto').randomBytes(32).toString('hex'))"`).
    - `BOOTSTRAP_ADMIN` — первый пользователь, создаётся при старте, если такого ещё нет.
      Формат `логин:пароль` или `логин:пароль:роль` (роли: `sysadmin` / `coder` / `analyst` / `tester` / `tourist`,
      по умолчанию `sysadmin` — чтобы было кому управлять пользователями). **Без этого войти будет некем.**
      Пример: `BOOTSTRAP_ADMIN=ivan:Qw7zP2k9x:sysadmin`. Защита от локаута: если ни у кого не осталось права
      `user.manage`, при старте этот пользователь автоматически повышается до `sysadmin`.
    - `AUTH_COOKIE_SECURE=1` — выставлять флаг `Secure` на cookie (включить на https).
  - **Ретенция чатов:** `CHAT_RETENTION_MS` (окно хранения истории, дефолт 7 дней) и
    `CHAT_RETENTION_SWEEP_MS` (период фоновой чистки, дефолт 1 час).
  - **Telegram (опционально):** `TELEGRAM_BOT_TOKEN` — токен от @BotFather (без него бот не запускается);
    `TELEGRAM_ALLOWED_CHAT_IDS` — разрешённые `chat_id` через запятую (узнать свой id — написать боту сообщение).
- **frontend/.env**
  - По умолчанию фронт ходит к бэкенду **через тот же origin** — Vite проксирует `/api` и `/ws` на бэкенд (`vite.config.ts`). Работает и на http, и на https, без mixed-content. Править обычно не нужно.
  - `VITE_BACKEND_HOST` / `VITE_BACKEND_PORT` — для прямого подключения к бэкенду в обход прокси.
    > ⚠️ Сессия хранится в httpOnly-cookie. Прямое подключение на **другой** хост/порт — это кросс-origin,
    > браузер может не слать cookie. Для авторизации держи фронт и бэкенд на одном origin (режим прокси по умолчанию).

## Запуск (dev)

В двух терминалах из корня репозитория:

```bash
npm run dev:backend     # tsc + node dist/index.js  → http://localhost:3001  (HTTP + WebSocket)
npm run dev:frontend    # Vite dev-сервер           → откроет адрес в выводе
```

Открой адрес, который покажет Vite (по умолчанию `localhost:5173`), и **войди** под пользователем
из `BOOTSTRAP_ADMIN`.

> ⚠️ **Бэкенд авто-перезагрузки не имеет** — он запускается из скомпилированного `dist/`.
> После любой правки в `backend/src/**` нужно заново `npm run dev:backend` (он делает `tsc` и перезапуск).
> Фронтенд на Vite подхватывает изменения по HMR — перезапуск не нужен.

### HTTPS и доступ с телефона (для PWA)

Service worker и установка PWA требуют secure-context. На `localhost` он есть всегда — поэтому
для разработки на своей машине ничего настраивать не надо. Чтобы открыть/установить приложение
**с телефона по IP**, нужен доверенный HTTPS-сертификат:

```bash
# один раз — установить локальный корневой CA в систему/браузер
mkcert -install

# выпустить сертификат на localhost + ваш LAN-IP, положить в frontend/certs/
cd frontend && mkdir -p certs && cd certs
mkcert -key-file key.pem -cert-file cert.pem localhost 127.0.0.1 <ваш-LAN-IP>
```

Если в `frontend/certs/` есть `key.pem` и `cert.pem`, Vite поднимает dev/preview по **HTTPS** на всех
интерфейсах (`host: true`) — открывай `https://<LAN-IP>:5173`. Чтобы PWA установилось на телефоне,
на нём тоже должен быть доверен корневой CA (скопировать `frontend/certs/rootCA.pem` и установить
как доверенный сертификат). Без сертификатов dev поднимется по http (localhost остаётся secure-context).
На https не забудь `AUTH_COOKIE_SECURE=1` в `backend/.env`.

## Сборка (prod)

В проде отдельный Vite-сервер не нужен — **бэкенд сам раздаёт собранный фронт** на `:3001` вместе с API/WS:

```bash
npm start               # = npm run build (backend tsc + frontend vite build) + node backend/dist/index.js
# открывай IDE на http://<хост>:3001  (порт 5173 в проде свободен)
```

Под капотом: бэкенд при старте проверяет `frontend/dist` (переопределяется `FRONTEND_DIST`) и раздаёт его
со SPA-fallback; хэшированные ассеты — с `Cache-Control: immutable`, `index.html`/`sw.js` — `no-cache`.
Сборка оптимизирована: вендоры (`react`, `xterm`), панель чата (`react-markdown` + подсветка) и терминал —
отдельные lazy-чанки; начальная загрузка ≈90 КБ gzip, PWA-precache ≈1 МБ.

Альтернатива (раздельно): `npm run build && npm --prefix frontend run preview` — фронт через Vite preview, бэкенд `npm run start:backend`.

## E2E-тесты (Playwright)

```bash
npx playwright install --with-deps chromium   # один раз: браузер + системные либы (нужен sudo)
npm run test:e2e                               # гоняет e2e/ против http://localhost:3001 (или IDE_URL=...)
```

Проверяет лаунчер, открытие воркспейса и ключевое — что серверные сессии переживают
закрытие вкладки браузера и восстанавливаются в свежей сессии.

## Использование

### Вход, роли и права

IDE закрыта логином. Первый пользователь сидируется из `BOOTSTRAP_ADMIN` (см. выше).

**Права — пер-пользовательские.** У каждого пользователя свой набор разрешений; роль задаёт лишь
*шаблон* этих прав при создании, подпись в интерфейсе и поведение агента. Реальный гейт всегда
проверяет права **конкретного пользователя**, а не роль. Действия: `agent.run` (чат с агентом),
`terminal.open`, `git.commit`, `git.push`, `project.add`, `project.delete`, `chat.read`, `user.manage`.

Роли (шаблоны прав по умолчанию):

| Роль | Шаблон прав |
|---|---|
| **Сисадмин** (`sysadmin`) | всё, **включая управление пользователями** — единственная роль с `user.manage` |
| **Кодер** (`coder`) | чат-агент, терминал, git commit/push, добавление/удаление проектов (без управления юзерами) |
| **Аналитик** (`analyst`) | чат-агент в режиме анализа + чтение; без терминала, коммитов и пушей |
| **Тестировщик** (`tester`) | чат-агент + терминал + чтение; без коммитов/пушей |
| **Турист** (`tourist`) | только чтение (список проектов, история чатов); без запуска агентов и терминалов |

**Управление пользователями — только у сисадмина.** В лаунчере ему доступна страница **«Пользователи»**:
таблица (имя, фамилия, почта, роль); клик по строке открывает сайдпанель с профилем, выбором роли и
**чекбоксами доступного функционала** — права включаются/выключаются индивидуально, поверх шаблона роли.
Под капотом — CRUD `GET/POST/PATCH/DELETE /api/auth/users` (за правом `user.manage`).

Кнопки в интерфейсе скрываются по фактическим правам пользователя, но **источник истины — сервер**:
каждый чувствительный REST-эндпоинт и WS-действие проверяют права независимо. Поведение агента тоже
идёт от прав пользователя: при стандартном для роли наборе подмешивается ролевой скилл `role-<роль>`,
а если права настроены индивидуально (отличаются от шаблона) — агент получает явный список того, что
ему разрешено и запрещено, и следует ему, а не названию роли.

### Проекты и где они хранятся

Каждый проект — отдельная папка внутри `PROJECTS_DIR` (по умолчанию `<DATA_DIR>/projects`).
Все папки в этом каталоге **автоматически** становятся проектами и появляются в лаунчере —
регистрировать вручную не нужно.

На входе (URL без `?p=`) открывается **лаунчер**: список проектов, создание нового, добавление
репозитория, удаление проекта и вход в общего менеджера. Выбор проекта открывает его в **текущей**
вкладке браузера (`?p=<id>`); **другой проект** — кнопка `↗` (в топбаре) открывает лаунчер в
**новой вкладке браузера**. Внутри проекта вкладки чатов/терминалов работают в фоне и не выгружаются.

### Чат с агентом

В контексте выбранного проекта в панели **ДЕЙСТВИЯ** открывается **🤖 Агент** — нативный чат с
интерактивным `claude`, работающим в директории проекта. Агент сам анализирует задачу, пишет и
правит код, делает коммиты, запускает сборку и тесты. В чате:

- пузыри сообщений и **markdown-рендер** ответа с подсветкой кода (кнопка «копировать» у блоков);
- **чипы вызовов инструментов** (`🔧 Edit · src/foo.ts`) — видно, что агент делает;
- **токеновый стриминг** ответа, кнопка **«Стоп»** (прервать) и **«Новый диалог»** (сбросить контекст);
- меню **стандартных команд** (git status, diff, тесты, сборка, коммит) — набор фильтруется по правам пользователя.

Можно открыть несколько чатов. История диалога хранится на сервере, переживает переподключение и
перезапуск бэкенда (контекст — через `--resume`), и чистится по ретенции (старше 7 дней).
**Чаты приватны**: каждый видит и продолжает только свои диалоги — даже у пользователей с доступом
к одному проекту истории не пересекаются (сервер проверяет владельца на каждой операции).

### Общий менеджер

**🧭 Общий менеджер** — отдельный кросс-проектный воркспейс (`?p=overseer`, выбирается в лаунчере;
cwd = корень всех проектов). Он:

- видит **все** проекты сразу (структура, последние коммиты, выжимка README подкладываются в контекст);
- отвечает на общие вопросы — что в каком проекте делается, как они связаны, что делать дальше;
- сам код не правит — при необходимости изменений **рекомендует открыть агента** нужного проекта;
- умеет добавлять репозитории по ссылке (см. ниже).

### Git

В панели **ДЕЙСТВИЯ** проекта: **Commit** (с вводом сообщения) и **Push** (по правам роли). Рядом —
лог последних коммитов, текущая ветка показана в топбаре. Просмотра/редактирования файлов в браузере
нет — это делает агент в чате.

### Добавление репозитория по ссылке

Два способа, оба клонируют в `PROJECTS_DIR/<имя>` (имя — из репозитория, в kebab-case):

1. **Кнопка** `➕ Добавить репозиторий` — ввести URL, бэкенд сделает `git clone` и зарегистрирует проект.
2. **Разговором** — попросить общего менеджера: «добавь репозиторий `<url>`» (он использует навык `add-repository`).

После клонирования проект сразу появляется в списке. Для приватных репозиториев нужен доступ
(SSH-ключ или токен) на машине, где работает бэкенд.

### Терминалы

**⌨ Терминал** — интерактивный bash-PTY в директории проекта (можно открыть несколько).
**Сессии живут на сервере** и привязаны к воркспейсу: при открытии проекта фронт запрашивает
`GET /api/workspaces/:wid/chats` и `.../terminals` и переподключается ко всем живым сессиям — даже в
свежей вкладке браузера с пустым localStorage. PTY держится живым при разрыве WS (буфер вывода
переотдаётся при reattach); закрытие вкладки гасит сессию сразу, отвязанные надолго — собираются GC (30 мин).

> 📡 Приложения, которые агент/терминал запускает **внутри проекта** (dev-серверы и т.п.), должны
> биндиться на `0.0.0.0` и открываться по сетевому IP машины (не localhost — IDE на сервере).
> Это правило автоматически доставляется агентам через `backend/PROJECTS_CLAUDE.md` → `PROJECTS_DIR/CLAUDE.md`.

### Скиллы агентов

Навыки лежат в репозитории в `backend/skills/` (формат `.claude/skills`):

- `add-repository` — клонирование репозитория по URL (для общего менеджера);
- `inspect-projects` — обзор проектов и рекомендации, какого агента открыть;
- `role-coder` / `role-analyst` / `role-tester` / `role-tourist` — возможности под роль пользователя (сисадмин использует надстройку как у кодера).

При старте бэкенда они копируются в `<PROJECTS_DIR>/.claude/skills/`, откуда их подхватывает `claude` CLI.
Чтобы добавить навык — создай папку `backend/skills/<имя>/SKILL.md` и перезапусти бэкенд.

### Telegram-бот (опционально)

Если в `backend/.env` задан `TELEGRAM_BOT_TOKEN`, при старте поднимается бот, дающий доступ к тем же
агентам и проектам из Telegram (агент работает в headless-режиме `claude -p`, контекст диалога держится
через `--resume`). Доступ — только для `chat_id` из `TELEGRAM_ALLOWED_CHAT_IDS`. Команды:

- `/agent` — выбрать роль (Агент / Общий менеджер);
- `/projects` — выбрать проект;
- `/status` — текущая привязка;
- `/reset` — сбросить контекст диалога;
- `/cancel` — прервать текущий ответ;
- `/help` — справка.
