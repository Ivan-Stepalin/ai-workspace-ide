# План: убрать редактор кода и упростить агентов + нативный чат с Claude

Статус: черновик-план (обсуждение перед реализацией).
Дата: 2026-06-06.

## 1. Цель и итоговое видение

Сейчас IDE — это «браузерный VS Code»: файловое дерево, Monaco-редактор, терминалы,
git и четыре роли агентов (manager / coder / reviewer / overseer). Мы меняем продукт:
**IDE перестаёт быть редактором кода и становится «пультом управления агентами»**.

Итоговая модель после рефакторинга:

- **Нет редактирования и просмотра кода в браузере.** Никакого Monaco, файлового дерева,
  вкладок-файлов. Если агенту нужно поправить код — он делает это сам внутри своей
  рабочей директории (он и так работает с ФС напрямую через инструменты `claude`).
- **Две роли вместо четырёх:** `manager` (внутри проекта) и `overseer` (общий менеджер
  над всеми проектами). `coder` и `reviewer` удаляются — их функции выполняет сам агент,
  работающий в кодовой базе.
- **Главный способ общения с агентом — нативный чат**, а не сырой PTY-терминал.
  Удобный UI с пузырями сообщений, рендером markdown, отображением вызовов инструментов
  и стримингом ответа. Сырой терминал (`bash`) остаётся как вспомогательный инструмент.
- **Появляется авторизация и ролевая модель пользователей** (§11): кодер / аналитик /
  тестировщик / турист. Под каждую роль — свой скилл агента с набором возможностей.
- **Чисто-фронтовые/инфраструктурные улучшения:** ретенция истории чатов (§8), перевод
  стилей на CSS Modules (§9), панель стандартных команд в чате (§10).

> ⚠️ **Две разные «роли», не путать.** (1) *Технический тип агента* (`manager`/`overseer`) —
> определяет рабочую директорию `claude` (проект vs `PROJECTS_DIR`). (2) *Роль авторизации
> пользователя* (кодер / аналитик / тестировщик / турист, §11) — определяет, **что** пользователю
> разрешено и **какой скилл** получает агент. Это независимые оси: тип агента = «где работаем»,
> роль пользователя = «что можно и с какими возможностями».

Что **остаётся**: проекты и их автообнаружение, git (log / branches / commit / push),
overseer с навыками (add-repository / inspect-projects), Telegram-бот, переподключение
к живым серверным сессиям, модель «один воркспейс = одна вкладка браузера».

## 4. Нативный чат с Claude вместо терминала (главная новая фича)

### 4.1 Проблема текущего подхода (PTY + xterm)

Сейчас агент = интерактивный `claude` в PTY, отрисованный через xterm. Минусы для чата:
- Это полноценный TUI: escape-коды, перерисовки, рамки — тяжело и шумно для «диалога».
- Нельзя аккуратно показать историю сообщений, отрендерить markdown, оформить блоки кода.
- Нет структуры «вопрос → ответ»: всё одна простыня терминала.
- На мобильном PTY-терминал особенно неудобен.

Терминал отлично подходит для ручной работы (`bash`), но плохо — для беседы с агентом.

### 4.2 Идея

Использовать **headless-режим `claude`** ровно как уже сделано в `telegram.ts`:

```
claude -p "<текст>" \
  --append-system-prompt "<роль>" \
  --output-format stream-json --verbose \
  --dangerously-skip-permissions \
  [--resume <session_id>]
```

`claude` отдаёт в stdout поток NDJSON-событий, мы парсим их и превращаем в события чата,
которые шлём в браузер по WebSocket. Контекст диалога держим через `--resume <session_id>`
(session_id берём из события `system/init`). `telegram.ts` — готовый референс этой логики
(`runAgent` + `handleEvent`, строки 81–156).

### 4.3 Типы событий stream-json (что парсим)

- `{ type:'system', subtype:'init', session_id }` → сохранить `session_id` для `--resume`.
- `{ type:'assistant', message:{ content:[...] } }` → блоки `text` (ответ) и `tool_use`
  (вызовы инструментов: имя + `input.file_path`/`command`/`pattern`).
- `{ type:'user', message:{ content:[{ type:'tool_result', ... }] } }` → результаты
  инструментов (можно показывать свёрнуто).
- `{ type:'result', result:'<финальный текст>' }` → итоговый ответ.
- Для **токенового стриминга** добавить флаг `--include-partial-messages` — тогда придут
  `{ type:'stream_event', event:{ type:'content_block_delta', delta:{ text } } }` и можно
  печатать ответ по мере генерации (как «печатает…»). Без него ответ приходит целиком
  одним `result`.

### 4.4 WebSocket-протокол (новые сообщения)

Чат живёт в **отдельном WS-соединении на вкладку** (как сейчас терминалы), чтобы не мешать
главному соединению бродкастов. Новые типы в `types.ts`:

Клиент → сервер:
- `chat_create` — `{ type:'chat_create', chatId, projectId, agent }` — открыть/переподключить
  чат-сессию. Ответ `chat_ready` + (если есть история) `chat_restore`.
- `chat_send` — `{ type:'chat_send', chatId, text }` — отправить сообщение агенту.
- `chat_cancel` — `{ type:'chat_cancel', chatId }` — прервать текущий ответ (kill процесса).

Сервер → клиент:
- `chat_ready` — `{ chatId }`.
- `chat_event` — `{ chatId, event }`, где `event` — нормализованное событие:
  `{ kind:'user', text }` | `{ kind:'tool', name, arg }` |
  `{ kind:'delta', text }` (если включён partial) | `{ kind:'assistant', text }` (финал) |
  `{ kind:'error', text }` | `{ kind:'done' }`.
- `chat_restore` — `{ chatId, messages:[...] }` — история при переподключении.

### 4.5 Backend: модуль `chat.ts`

Новый файл `backend/src/chat.ts` (или секция в `index.ts` рядом с `terminal_create`):

- Хранилище сессий чата на сервере: `Map<chatId, ChatSession>`, где
  `ChatSession = { projectId, agent, sessionId, history: Msg[], proc: ChildProcess|null, ws }`.
  Источник истины — сервер (как у терминалов), чтобы переподключение из свежей вкладки
  восстанавливало диалог. `sessionId` + краткую историю персистим в `workspace.db`
  (новая таблица `chat_sessions`, по образцу `tg_sessions`), чтобы переживало рестарт бэкенда.
- `chat_send`: собрать `cwd` (как в `terminal_create`: overseer → `PROJECTS_DIR`,
  иначе `getProject().path`), `spawn('claude', [...])` со `stream-json`, распарсить NDJSON
  (взять `handleEvent` из `telegram.ts`, обобщив колбэк), складывать события в историю и
  пушить `chat_event` в `ws`. На `exit` — `chat_event {kind:'done'}`.
- Один активный процесс на чат (как `running[chatId]` в telegram). Повторный `chat_send`
  при занятости — ошибка/очередь.
- `chat_cancel` — `proc.kill()`.
- GC отсоединённых сессий по образцу `DETACH_GC_MS` терминалов.
- Переиспользовать parsing-логику: вынести `handleEvent`/`spawn`-обвязку в общий хелпер,
  который зовут и `telegram.ts`, и `chat.ts` (убрать дублирование).

`GET /api/workspaces/:wid/chats` — список живых чат-сессий воркспейса для переподключения
(аналог существующего `/terminals`).

### 4.6 Frontend: `ChatPanel.tsx`

Новый lazy-компонент (как `Terminal.tsx`), обёрнут в `memo`:

- Своё WS-соединение, шлёт `chat_create` с `chatId` (= `wsId` вкладки), затем `chat_send`.
- Состояние `messages: ChatMessage[]`; по `chat_event` дополняет последнее сообщение
  ассистента (`delta`) или добавляет новое.
- Рендер:
  - пузыри пользователь/ассистент;
  - **markdown** ответа ассистента (добавить `react-markdown` + `remark-gfm`, подсветка
    кода — лёгкий `highlight.js`/`shiki` или просто `<pre>`); блоки кода с кнопкой «копировать»;
  - **чипы вызовов инструментов** (`🔧 Edit · src/foo.ts`, `🔧 Bash · npm test`) —
    свёрнутые строки между сообщениями, разворачиваются по клику;
  - индикатор «агент печатает…» во время стрима, кнопка «Стоп» (→ `chat_cancel`);
  - поле ввода снизу: Enter — отправить, Shift+Enter — перенос; автоскролл вниз.
- Кнопка «Новый диалог» (сброс контекста → новый chatId / `--resume` сбрасывается).

**App.tsx интеграция:**
- Новый `Tab` вариант: `{ uid; type:'chat'; agentType; num; wsId }`.
- `openAgent()` теперь открывает **чат-вкладку** (`type:'chat'`), а не agent-терминал.
- Кнопки в правой панели: «🤖 Агент» (manager → чат), для overseer «🧭 Общий менеджер» (чат).
- Сырой `bash`-терминал остаётся отдельной кнопкой «⌨ Терминал» (тип `terminal`).
- Переподключение: к `/terminals` (bash) добавить `/chats` (чаты).

### 4.7 Сравнение и решение

| Критерий | PTY-терминал (сейчас) | Нативный чат (предлагается) |
|---|---|---|
| Удобство диалога | низкое | высокое |
| Рендер markdown/кода | нет | да |
| Стриминг | да (сырой) | да (через partial events) |
| Видны вызовы инструментов | да (шумно) | да (структурно, чипами) |
| Интерактивные prompt’ы claude | да | нет (но мы и так `--skip-permissions`) |
| Мобильный UX | плохой | хороший |
| Сложность реализации | есть | средняя (есть референс в telegram.ts) |

**Решение:** делаем нативный чат основным интерфейсом агента. Сырой PTY-терминал
сохраняем как кнопку «Терминал» для ручных команд. Так мы не теряем интерактивный
`claude` совсем — при желании его можно вернуть отдельной кнопкой «Агент в терминале».

## 8. Ретенция истории чатов (удаление сообщений старше недели)

История чатов не должна копиться бесконечно — храним только последние 7 дней.

**Хранилище.** Таблица из §4.5 расширяется отдельной таблицей сообщений (вместо хранения
истории одним JSON-блобом — так чистка по дате тривиальна):

```sql
CREATE TABLE chat_messages (
  id        INTEGER PRIMARY KEY AUTOINCREMENT,
  chat_id   TEXT NOT NULL,
  role      TEXT NOT NULL,          -- 'user' | 'assistant' | 'tool'
  content   TEXT NOT NULL,
  created_at INTEGER NOT NULL       -- unix-ms
);
CREATE INDEX idx_chat_messages_created ON chat_messages(created_at);
```

**Чистка.** В `chat.ts` — функция `pruneOldMessages()`:

```ts
const WEEK_MS = 7 * 24 * 60 * 60 * 1000;
db.prepare('DELETE FROM chat_messages WHERE created_at < ?').run(Date.now() - WEEK_MS);
```

Вызываем: (1) при старте бэкенда, (2) по таймеру `setInterval` раз в час (период — env
`CHAT_RETENTION_SWEEP_MS`, дефолт 1ч; окно — env `CHAT_RETENTION_MS`, дефолт 7 дней).
Пустые чат-сессии (без сообщений за окно) тоже удаляем из `chat_sessions`, а их `session_id`
для `--resume` теряет смысл — следующий `chat_send` начнёт новый диалог.

> Опционально применить ту же ретенцию к `tg_sessions`/истории Telegram, если будем хранить
> там историю. Сейчас Telegram историю не персистит — только `session_id`, его не трогаем.

## 9. Перевод стилей на CSS Modules

Сейчас фронт на **Tailwind v4** (`@import "tailwindcss"` + `@theme`-токены, утилиты в `className`,
динамика — инлайн `style`). Переводим на **CSS Modules** (`*.module.css`, скоупленные классы).

**Подход:**
- Палитру из `index.css` `@theme` сохранить как обычные CSS-переменные в `:root`
  (глобальный `theme.css` или `index.css`): `--color-app`, `--color-sidebar`, `--color-accent`
  и т.д. — они продолжат работать и в модулях через `var(--color-…)`.
- На каждый компонент — свой `Component.module.css`; строки утилит Tailwind в `className`
  заменить на семантические классы (`styles.topbar`, `styles.tab`, `styles.tabActive`).
- Условные классы — через хелпер склейки (`clsx` или ручной join), вместо длинных
  шаблонных строк Tailwind.
- Динамические цвета (по роли/статусу) — остаются инлайновым `style`, как и сейчас.
- Убрать `@tailwindcss/vite` из `vite.config`, удалить `@import "tailwindcss"`; зависимости
  `tailwindcss`/`@tailwindcss/vite` из `package.json`.

**Объём.** Это самый «механический», но широкий по охвату пункт: затрагивает **все** компоненты
(`App`, `ChatPanel`, `Terminal`, `ProjectPicker`, `AddRepoModal`, `ConfirmModal`, `PromptModal`).
Делать **по одному компоненту за коммит**, тема-переменные ввести первым шагом, чтобы Tailwind
и модули какое-то время сосуществовали.

> Решение: переменные палитры — общий `theme.css`; склейка классов — `clsx`; миграция
> покомпонентно. Обновить раздел «Стили» в `CLAUDE.md` (Tailwind → CSS Modules).

## 10. Чат: кнопки стандартных команд (в селекте)

В `ChatPanel` рядом с полем ввода — компактный **селект/дропдаун стандартных команд**
(чтобы не загромождать UI кнопками). Выбор пункта либо вставляет команду в поле ввода,
либо сразу отправляет.

**Содержимое селекта (черновой набор):**
- Управление диалогом: «Новый диалог» (сброс `--resume`), «Сжать контекст», «Повторить».
- Частые задачи: «Запустить тесты», «Собрать проект», «git status», «Покажи изменения (diff)»,
  «Закоммить изменения».
- Команды `claude` при необходимости (`/clear`, `/compact`) — если уместно в headless-режиме.

**Реализация.**
- Список команд — конфиг-массив `{ id, label, prompt, send?: boolean }`.
- **Набор команд зависит от роли пользователя** (§11): туристу — только безопасные/read-only
  (status, diff), кодеру — полный набор (коммит, сборка), и т.д. Источник — capabilities роли.
- UI: `<select>`/кастомный дропдаун в строке ввода; выбор → `insert` в textarea или
  немедленный `chat_send`.

## 11. Ролевая модель с авторизацией + скиллы под роли

Сейчас авторизации нет вообще — IDE открыта на `:3001` без логина. Вводим **аутентификацию**
и **роли пользователей**, и привязываем к каждой роли **скилл агента** с набором возможностей.

### 11.1 Роли

| Роль | Назначение | Возможности (детали — позже) |
|---|---|---|
| **Кодер** | пишет/правит код | полный доступ: чат-агент, терминал, git commit/push, все команды |
| **Аналитик** | исследует, планирует | чтение кода, чат-агент в режиме анализа, git read-only, без коммитов |
| **Тестировщик** | проверяет, гоняет тесты | запуск тестов/сборки, чтение, чат-агент с тест-скиллом, без правок прод-кода |
| **Турист** | гость, только смотрит | read-only: список проектов, история чатов; без запуска агентов/команд |

> Точные capability-матрицы под каждую роль опишем отдельно — здесь фиксируем каркас.

### 11.2 Аутентификация

- **Хранилище пользователей** в `workspace.db`:
  ```sql
  CREATE TABLE users (
    id        TEXT PRIMARY KEY,
    username  TEXT UNIQUE NOT NULL,
    pass_hash TEXT NOT NULL,      -- bcrypt/argon2
    role      TEXT NOT NULL,      -- 'coder' | 'analyst' | 'tester' | 'tourist'
    created_at INTEGER NOT NULL
  );
  ```
- **Логин:** `POST /api/auth/login` → проверка пароля → выдать сессионный токен
  (JWT в httpOnly-cookie или подписанный session-id; cookie проще для SPA). `POST /api/auth/logout`,
  `GET /api/auth/me` (текущий пользователь + роль).
- **Защита API:** middleware на все `/api/*` (кроме login) — проверяет токен, кладёт
  `req.user = { id, role }`. **Защита WebSocket:** при `subscribe`/`terminal_create`/`chat_create`
  проверять токен (передавать его первым сообщением или query-параметром при апгрейде).
- **Первый пользователь / сидинг:** env `BOOTSTRAP_ADMIN` или CLI-команда создания юзера;
  пароли только хешами, секрет для токена — env `AUTH_SECRET` (с дефолтом-предупреждением).

### 11.3 Авторизация (gate по capabilities)

- Единый модуль `auth/permissions.ts`: `can(role, action)` где `action` —
  `'agent.run' | 'terminal.open' | 'git.commit' | 'git.push' | 'project.add' | 'chat.read' | …`.
- **Backend** проверяет `can()` в каждом чувствительном хендлере (запуск чата/терминала, commit,
  push, add-repo, delete). Турист на запрещённом действии получает 403 / WS-ошибку.
- **Frontend** прячет/дизейблит кнопки по роли (источник — `GET /api/auth/me`): туристу не
  показываем «Терминал»/«Агент»/commit; набор команд в селекте (§10) фильтруется по роли.

### 11.4 Скилл агента под роль

Каждой роли соответствует **скилл** в формате `.claude/skills` (как существующие
`backend/skills/<name>/SKILL.md`, которые `syncManagerSkills()` раскладывает в
`PROJECTS_DIR/.claude/skills`):

```
backend/skills/
  role-coder/SKILL.md        # возможности кодера
  role-analyst/SKILL.md      # возможности аналитика
  role-tester/SKILL.md       # возможности тестировщика
  role-tourist/SKILL.md      # минимум (или вовсе без агента)
```

- При запуске чата/агента подмешиваем нужный скилл и/или роль-специфичный системный промпт
  в зависимости от `req.user.role`: расширяем `PROMPTS`/`--append-system-prompt` ролевой
  частью, а скилл становится доступен агенту через каталог skills.
- Содержимое `SKILL.md` (конкретные инструменты, ограничения, чек-листы) — **опишем позже**,
  здесь фиксируем структуру и точку интеграции.
- `syncManagerSkills()` обобщить, чтобы синхронизировать и роль-скиллы.

### 11.5 Связь с остальным планом

- **Чат (§4):** `cwd` и тип агента не меняются, но к системному промпту добавляется ролевая
  надстройка + скилл; набор стандартных команд (§10) фильтруется ролью.
- **Telegram (§3):** маппинг telegram chat_id → роль (поле `role` в `tg_sessions` или связь
  с `users`), те же `can()`-проверки. Деталь второго приоритета.
- **Турист и чтение:** ему доступна история чатов (с учётом ретенции §8), но не запуск агентов.
